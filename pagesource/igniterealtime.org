                Posted by <a href="https://discourse.igniterealtime.org/u/Dele_Olajide">Dele Olajide</a></span><span class="ignite_blog_entry_date">3 months, 5 days ago</span></div><div class="ignite_blog_entry_body">
                <p><a href="https://discourse.igniterealtime.org/u/guus">Guus</a> has generously provided a new Openfire <a href="http://www.igniterealtime.org/projects/openfire/plugins/httpfileupload/readme.html" rel="nofollow noopener">plugin</a> that implements <a href="https://xmpp.org/extensions/xep-0363.html" rel="nofollow noopener">XEP 0363</a> and shows off the flexibility of having the Jetty web server embedded in Openfire. If you want to perform file upload/download from the Candy Chat web application and other Strophe based web applications, then take a look at this Candy Chat plugin.</p>
<p><div class="lightbox-wrapper"><a class="lightbox" href="https://discourse.igniterealtime.org/uploads/default/original/2X/4/44ba39c038f89fd9ba326014e8587395ceceb5f3.png" data-download-href="https://discourse.igniterealtime.org/uploads/default/44ba39c038f89fd9ba326014e8587395ceceb5f3" title="image.png"><img src="https://discourse.igniterealtime.org/uploads/default/optimized/2X/4/44ba39c038f89fd9ba326014e8587395ceceb5f3_1_690x108.png" width="690" height="108"><div class="meta">
<span class="filename">image.png</span><span class="informations">1456x228 33.5 KB</span><span class="expand"></span>
</div></a></div></p>
<p>You can either drag and drop the files into the conversation window directly or click on the paperclip icon in the toolbar to select files. It then does the following:</p>
<ul>
<li>Requests for an upload/download pair of URLs (called a slot) from the Openfire httpfileupload plugin</li>
<li>Does an HTTP PUT request to upload the file using the given slot</li>
<li>Pastes the slot?s GET url into the room conversation and converts it into a clickable link on receipt.</li>
</ul>
<p>The <a href="https://github.com/igniterealtime/Pade/tree/master/extension/groupchat/candyshop/upload" rel="nofollow noopener">candy upload plugin</a> is now available in Pàdé  0.1.15.</p>
<h2>Update</h2>
<p>You can also upload/download from a video-conference using Jitsi Meet in Pàdé. Only drag and drop is supported for uploading. Drag the file and drop anywhere on the conversation panel to upload the file. When the load is uploaded, a clickable link appears in the conversation.</p>

                            <div class="ignite_blog_entry "><div class="ignite_blog_entry_header"><img height="32" width="32" alt="Dele Olajide" src="https://discourse.igniterealtime.org/user_avatar/discourse.igniterealtime.org/dele_olajide/32/5369_1.png"/><div class="ignite_blog_entry_comments"><a href="https://discourse.igniterealtime.org/t/openfire-meetings-desktop-screen-sharing-using-pade">0</a></div><h2><a title="Openfire Meetings Desktop/Screen Sharing using Pàdé" href="https://discourse.igniterealtime.org/t/openfire-meetings-desktop-screen-sharing-using-pade">Openfire Meetings Desktop/Screen Sharing using Pàdé</a></h2><span class="ignite_blog_entry_author">
                Posted by <a href="https://discourse.igniterealtime.org/u/Dele_Olajide">Dele Olajide</a></span><span class="ignite_blog_entry_date">3 months, 1 week ago</span></div><div class="ignite_blog_entry_body">
                <p>Screen sharing with Openfire Meetings is a bit tricky no thanks to Google?s WebRTC security policy to use Chrome extensions for trust. This means you have to <a href="https://medium.com/@chris_82106/implementing-webrtc-screen-sharing-in-a-web-app-late-2016-51c1a2642e4" rel="nofollow noopener">develop a personalised chrome extension</a> to white-list your domain as trusted. I don?t use Firefox and consequently have no idea of the user experience.</p>
<p>With <a href="https://chrome.google.com/webstore/detail/pade-openfire-meetings/fohfnhgabmicpkjcpjpjongpijcffaba" rel="nofollow noopener">Pàdé version 0.1.13</a>, I have tried to improve the experience as follows:</p>
<ul>
<li>
<p>If you use ofmeet from Pàdé exclusively, then you can screen/desk share from any domain. No extra work involved.</p>
</li>
<li>
<p>If you still use the <a href="https://github.com/igniterealtime/community-plugins/tree/master/ofmeet" rel="nofollow noopener">ofmeet 0.3.x version</a>, then you can use Pàdé to provide screen/desk-share from any domain as well. You should check ?Desktop Share mode only? in the options screen</p>
</li>
<li>
<p><div class="lightbox-wrapper"><a class="lightbox" href="https://discourse.igniterealtime.org/uploads/default/original/2X/7/73179df097ec2d8d351a36dab9debd543ad3615d.png" data-download-href="https://discourse.igniterealtime.org/uploads/default/73179df097ec2d8d351a36dab9debd543ad3615d" title="image.png"><img src="https://discourse.igniterealtime.org/uploads/default/optimized/2X/7/73179df097ec2d8d351a36dab9debd543ad3615d_1_522x499.png" width="522" height="499"><div class="meta">
<span class="filename">image.png</span><span class="informations">776x743 25.3 KB</span><span class="expand"></span>
</div></a></div></p>
</li>
<li>
<p>If you are using the <a href="https://github.com/igniterealtime/ofmeet-openfire-plugin" rel="nofollow noopener">ofmeet latest 0.9.x version</a>, then you have to <a href="https://github.com/igniterealtime/Pade/blob/master/extension/manifest.json" rel="nofollow noopener">white-list your domain</a> in order for Pàdé to support it. To do this, you can either raise a PR on the Pàdé repository at GitHub or raise an issue and I will add it manually. When the next Pàdé release is made, your domain should be supported.</p>
</li>
</ul>
<p>Please note that <strong>your domain name will be exposed</strong>. This should not be a problem for a private network, but could expose your server if your domain is on the Internet. In that case, write your own chrome extension for Openfire Meetings or pay someone to do it or simply use Pàdé <img src="https://discourse.igniterealtime.org/images/emoji/twitter/slight_smile.png?v=5" title=":slight_smile:" class="emoji" alt=":slight_smile:"></p>

                Posted by <a href="https://discourse.igniterealtime.org/u/akrherz">daryl herzmann</a></span><span class="ignite_blog_entry_date">3 months, 1 week ago</span></div><div class="ignite_blog_entry_body">
                <p>The Ignite Realtime Community is thrilled to announce the availability of Openfire 4.2.1.  <strong>But wait</strong>, you may wonder why Openfire 4.2.0 was not announced nor blogged about.  The answer is that after a soft release of Openfire 4.2.0, we noticed a number of folks hitting a <a href="https://discourse.igniterealtime.org/t/openfire-4-2-0-login-issues-check-your-database-schema/79916">database schema issue</a> that would cause logins to fail. We decided to turn around a fix for that issue and now fully announce the new release.</p>
<p>So what has changed with Openfire 4.2?  A lot! Some highlights:</p>
<ul>
<li>XEP-0237 Roster Versioning</li>
<li>Server to Server (s2s) is more robust</li>
<li>The websocket functionality, previously in a plugin, was merged into core</li>
<li>The Admin Console now has a Pub-Sub administration interface</li>
<li>You can now manually test a Server to Server connection on the Admin Console</li>
<li>XEP-0198 Client Resumption is now available</li>
<li>A lot of polish and new handy functionality was added to the Admin Console</li>
<li>Openfire plugin loading is much more robust</li>
</ul>
<p>The <a href="http://download.igniterealtime.org/openfire/docs/latest/changelog.html">changelog</a> denotes the full listing of Jira issues addressed in this release and 4.2.0.</p>
